QosRates Values
 
Table 71: QosRates Values  
QOS Rate
DESCRIPTION
UNSPECIFIED
rate is unspecified. Typically used by QoS initialization methods and not intended to be encoded or decoded.
TICK_BY_TICK
rate is tick-by-tick (i.e., data is sent for every update). This is the highest quality rate value. The best overall QoS is a tick-by-tick rate with a timeliness of QosTimeliness.REALTIME.
JIT_CONFLATED
rate is Just-In-Time (JIT) Conflated, meaning that quality is typically tick-by-tick, but if a data burst occurs (or if a component cannot keep up with tick-by-tick delivery), multiple updates are combined into a single update to reduce traffic. This value is usually considered a lower quality than TICK_BY_TICK.
Because JIT conflation is triggered by an applicationâ€™s inability to keep up with data rates, the effective rate depends on whether the application can sustain full data rates.
Use of this value typically results in a rate similar to TICK_BY_TICK. However, when the application cannot keep up with data rates, it results in a rate similar to TIME_CONFLATED, where rateInfo is determined by the provider. Specific information about conflationTime or conflationCount might be present in an UpdateMsg. For further details, refer to Section 12.2.3.
TIME_CONFLATED
rate is time-conflated. The interval of time (usually in milliseconds) over which data are conflated is provided in Qos.rateInfo. This is lower quality than TICK_BY_TICK and at times even lower than JIT_CONFLATED. Specific information about the conflationTime or conflationCount might be present in the UpdateMsg. For more details, refer to Section 12.2.3.
