UPANIProvider_Module_1a_Connect
The first step of a non-interactive provider application is to establish network communication with an ADH server by creating an outbound connection to the well-known hostname and port on the ADH. The non-interactive provider uses the Transport.connect() method to initiate the connection process and uses the Channel.init() method to complete channel initialization.
UPANIProvider_Module_1b_Ping
Ping (i.e., heartbeat) messages indicate the continued presence of an application. After establishing a connection, ping messages must be exchanged. The negotiated ping timeout is retrieved using the Channel.info() method. If heartbeats are not sent or received within a specified time frame, the connection is terminated.
.
UPANIProvider_Module_1c_ReadWrite
When channel initialization is complete, the state of the channel (ChannelState) is ACTIVE. At this time, an application can send or receive data.
UPANIProvider_Module_2_Login
Applications authenticate using the Login domain model. An OMM non-interactive provider must authenticate with an ADH using a login request prior to providing content. After receiving a login request, an ADH determines whether a user has permissions to access the system. The ADH sends back a login response, indicating whether the Non-Interactive Provider is granted access.
UPANIProvider_Module_3_ProvideDirectory
The Source Directory domain model conveys information about all available services in the system. An OMM non-interactive provider application must provide Source Directory information, specifying all available services, including information about supported domain types, the service’s state, the quality of service (QoS), and any associated item group information.
UPANIProvider_Module_4_Dictionary
Applications often require a dictionary to encode or decode data. This dictionary typically defines type and formatting, and directs the application in encoding or decoding data. Content that uses the FieldList type requires the use of a field dictionary (usually RDMFieldDictionary, though it can also be user-defined or a modified field dictionary).
The OMM non-interactive provider will use locally available dictionaries (i.e., defined in a file).
UPANIProvider_Module_5_ProvideContent
After providing a Source Directory, the OMM non-interactive provider application can begin pushing content to the ADH. This example illustrates sending one Market Price domain item refresh, update, and close status message(s) to an ADH.
