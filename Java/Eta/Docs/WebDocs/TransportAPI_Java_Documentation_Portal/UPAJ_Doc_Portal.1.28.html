Transport API Java Edition

Consumer and Provider Examples
Consumer
This single-threaded client application consumes or posts content between an OMM consumer and OMM provider. The application observes the following application flow:
•
Initialize the Transport API transport and connect the client.
•
Send a login request, a source directory request, a dictionary request, and one or more item request messages (snapshot or streaming) to a provider
•
Process the responses and display them in the console.
This application supports consuming Level I Market Price, Level II Market By Order, Level II Market By Price, and Yield Curve information as normal or private streams.
NIProvider
This single-threaded client application non-interactively provides Level I Market Price and Level 2 Market By Order data to an Advanced Data Hub (ADH). The application observes the following application flow:
•
Initialize the Transport API transport and connect the client.
•
Load the dictionary from a file. If the dictionary cannot be loaded, the application exits.
•
Send a login request like a consumer to the ADH and processes dictionary requests like a provider.
•
Source directory refresh messages and market price and/or market by order refresh/update messages are published to the ADH without any request for them.
Reliable multicast can be used to communicate between this application and any ADH on the network. The non-interactive provider can then send one message to all ADH's on the network instead of having to fan-out messages to each ADH TCP/IP connection.
Provider
This single-threaded server application interactively provides Level I Market Price, Level II Market By Order, Level II Market By Price, and Symbol List data to one or more consumers. The application observes the following application flow:
•
Initialize the Transport API transport and bind the server.
•
Load dictionary information from the RDMFieldDictionary and enumtype.def files.
•
Process login, source directory, dictionary, market price, market by order, market by price, and symbol list requests from consumers.
•
Sends appropriate responses.
Level II Market By Price refresh messages are sent as multi-part messages. An update message is sent between each part of the multi-part refresh message.
