Data Structures

struct  RsslRequestMsg
 The RsslRequestMsg is used to provide an initial or subsequent data synchronization point, including quality of service and payload content. When used to respond to an RsslRequestMsg, a solicited RsslRefreshMsg can be used. If pushing out a content change to downstream applications, an unsolicited RsslRefreshMsg can be used. When the stream is instantiated because of an RsslRequestMsg, the streamId from the request should be used; when a provider instantiates the stream with an RsslRefreshMsg (e.g. Non-Interactive provider), a negative streamId should be used. Enumeration Type Documentation

enum RsslRequestFlags
See also:
RsslRequestMsg, RsslMsg
Enumerator:
RSSL_RQMF_NONE 

(0x000) No RsslRequestMsg flags are present

RSSL_RQMF_HAS_EXTENDED_HEADER 

(0x0001) This RsslRequestMsg has an extended header buffer, contained in RsslRequestMsg::extendedHeader.

RSSL_RQMF_HAS_PRIORITY 

(0x0002) This RsslRequestMsg has priority information, contained in RsslRequestMsg::priorityClass and RsslRequestMsg::priorityCount. This is used to indicate the importance of this stream.

RSSL_RQMF_STREAMING 

(0x0004) Indicates that the RsslRequestMsg's item stream should be opened as streaming (e.g. as changes occur, subsequent RsslUpdateMsg will deliver updates). If a stream is paused, a subsequent RsslRequestMsg with this flag present will resume data flow.

RSSL_RQMF_MSG_KEY_IN_UPDATES 

(0x0008) Indicates that the user would like to have RsslMsgKey information included in any RsslUpdateMsg content on the stream.

RSSL_RQMF_CONF_INFO_IN_UPDATES 

(0x0010) Indicates that the user would like to have conflation information, when conflation is occuring, included in any RsslUpdateMsg content on the stream.

RSSL_RQMF_NO_REFRESH 

(0x0020) Indicates that the user does not require an RsslRefreshMsg for this request - typically used as part of a reissue to change priority, view information, or pausing/resuming a stream.

RSSL_RQMF_HAS_QOS 

(0x0040) This RsslRequestMsg contains quality of service information, contained in RsslRequestMsg::qos. If only RsslRequestMsg::qos is present, this is the QoS that will satisfy the request. If RsslRequestMsg::qos and RsslRequestMsg::worstQos are both present, this indicates that any QoS in that range will satisfy the request.

RSSL_RQMF_HAS_WORST_QOS 

(0x0080) This RsslRequestMsg contains a worst quality of service, implying that a range is specified in the message. When RsslRequestMsg::qos and RsslRequestMsg::worstQos are both present, this indicates that any QoS in that range will satisfy the request.

RSSL_RQMF_PRIVATE_STREAM 

(0x0100) Indicates that the user would like to open the stream as 'private'. This requires a response (RsslRefreshMsg, RsslStatusMsg, or RsslAckMsg) to acknowledge that it is open as private. Once opened as private, all content should only flow between two endpoints of streams - no fanout or caching should be performed.

RSSL_RQMF_PAUSE 

(0x0200) Indicates that the user would like to pause the stream. If present on an initial RsslRequestMsg, the stream should be opened in a paused state where the user gets an initial RsslRefreshMsg. If present on a subsequent RsslRequestMsg, the stream will be paused at that time. Issuing this flag does not guarantee that the stream will be paused.

RSSL_RQMF_HAS_VIEW 

(0x0400) Indicates that the payload contains a dynamic view specification, indicating the specific information on the stream the user is interested in receiving.

RSSL_RQMF_HAS_BATCH 

(0x0800) This RsslRequestMsg payload contains a batch of items for requesting

RSSL_RQMF_QUALIFIED_STREAM 

(0x1000) Indicates that the user would like to open the stream as 'qualified.'
