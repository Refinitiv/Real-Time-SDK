UPAProvider_Module_1a_Connect com/thomsonreuters/upa/training/provider
The first step of any interactive provider is to establish a listening socket on a well-known port so that consumer applications can easily connect. The provider uses the rsslBind() function to open the port and listen for incoming connection attempts.
Whenever an OMM consumer application attempts to connect, the provider uses the rsslAccept() function to start connection initialization.
In this training example, the interactive provider supports only a single client.

UPAProvider_Module_1b_Ping
Ping (i.e., heartbeat) messages indicate the continued presence of an application. After establishing a connection, ping messages must be exchanged. The negotiated ping timeout is retrieved using the rsslGetChannel() function. If heartbeats are not sent or received within the expected time frame, the connection is terminated.

UPAProvider_Module_2_HandleLogin
Applications authenticate using the Login domain model. An OMM interactive provider must handle the consumer’s login request message and supply an appropriate response. After receiving a login request, the Interactive Provider must perform authentication and permissioning.

UPAProvider_Module_3_ProvideDirectory
An OMM provider provides source directory information specifying available services and their capabilities. This includes information about supported domain types, the service’s state, the quality of service (QoS), and any associated item group information.

UPAProvider_Module_4_ProvideDictionary
Applications often require a dictionary to encode or decode data. This dictionary typically defines type and formatting information and the application uses it to encode or decode specific pieces of information. Content that uses the RsslFieldList type requires the use of a field dictionary (usually RDMFieldDictionary, though it can also be a user-defined or modified field dictionary).
Optionally, providers can provide dictionary information to consumers.
